# Phase 0 Research: Multi-Transport Server APIs

**Feature**: 003-rest-grpc-api | **Date**: 2026-06-20

Resolves every technical unknown and "NEEDS CLARIFICATION" from the plan's
Technical Context and the spec's Assumptions. Grounded in: MuninnDB
(`scrypster/muninndb` `docs/architecture.md`) as the reference architecture; the
existing go-rag codebase (read this session); and 2026 Go-ecosystem web research.

---

## R1. gRPC toolchain: grpc-go (decision — MuninnDB parity)

**Decision**: Use **grpc-go** (`google.golang.org/grpc`) for the gRPC transport,
with a single `proto/gorag.proto` code-generated by `buf`
(`protoc-gen-go` + `protoc-gen-go-grpc` → `proto/gen/*.pb.go` + `*_grpc.pb.go`).
This is the same toolchain MuninnDB uses (`proto/gen`, `proto/muninn`,
`internal/transport/grpc`), per the user's instruction to match MuninnDB.

**Rationale**:
- **MuninnDB parity**: identical gRPC stack and generated-code layout
  (`service` → `RegisterXxxServer`, `XxxClient`) — faithful to the reference
  architecture the spec is built on.
- **Pure-Go, statically linked**: grpc-go + `google.golang.org/protobuf` are
  pure-Go, Apache-2.0/BSD-3-Clause. `CGO_ENABLED=0` static build preserved
  (Principle III). `google.golang.org/protobuf` v1.33.0 is already in `go.mod`
  (indirect) — promoted to direct; `google.golang.org/grpc` is the one new
  runtime dependency.
- **Polyglot by default**: grpc-go speaks canonical gRPC over HTTP/2, so clients
  in any language (Python/Node/Java/Rust) interop with zero friction — exactly
  MuninnDB's model.
- **Mature/streaming-ready**: battle-tested, full interceptor ecosystem (bearer
  auth via a unary interceptor checking `authorization` metadata), native
  streaming if needed later. Unary-only for v1 (R5).

**Alternatives considered**:
- **connect-go** (`connectrpc.com/connect`) — gRPC-compatible, built on stdlib
  `net/http`, smaller than grpc-go, 2026-idiomatic. **Rejected** by user
  decision: match MuninnDB. (Would have been the lighter-weight default for a
  single-node local service; grpc-go's size is the accepted trade-off for parity.)
- **connect-go's Connect-JSON as the REST transport too** — one schema yields gRPC
  + JSON. Rejected regardless of gRPC choice: REST stays a separate, hand-written,
  OpenAPI-described, curl-friendly surface (R2/R4).

**Sources**: [grpc-go GitHub](https://github.com/grpc/grpc-go);
[MuninnDB `internal/transport/grpc`](https://github.com/scrypster/muninndb)
(`server.go`, `engine_adapter.go`); [connect-go](https://github.com/connectrpc/connect-go)
(alternative).

## R2. REST routing: stdlib `net/http` (decision)

**Decision**: REST is built on the **Go stdlib `net/http` ServeMux** (Go ≥1.22
pattern routing: `mux.HandleFunc("POST /v1/query", ...)`). No router framework.

**Rationale**: go-rag's REST surface is small (~11 resource-oriented endpoints).
Stdlib pattern-mux covers it with zero new dependencies — the strongest possible
fit for Principle III. `httptest` drives handler tests directly.

**Alternatives considered**:
- **`chi` / `gin` / `echo`** — rejected: add a dependency for routing the stdlib
  already handles; middleware needs (one bearer-token check) are trivial as a
  wrapping handler. Revisit only if cross-cutting middleware grows.

## R3. Authentication: reuse the existing local bearer token (resolves spec "no-auth" assumption)

**Decision**: REST and gRPC reuse go-rag's **existing local bearer token**
(`internal/daemon/pid.go:ReadToken`, file `mcp.token`), enforced identically to
MCP today. REST checks `Authorization: Bearer <token>`; gRPC/connect checks the
`authorization` metadata header. Empty token ⇒ no auth (development mode).

**Rationale**: The spec assumed "no auth, trusted-local." The codebase already
implements a local bearer token for MCP — reusing it across all three transports
is strictly better (consistent, already tested) and keeps go-rag trusted-local
(no multi-user/RBAC, no TLS, still loopback-only — fully within PRD §2.2 scope).

**Alternatives considered**:
- **No token on REST/gRPC** — rejected: inconsistent with the existing MCP guard.
- **Vault-scoped API keys + TLS (MuninnDB model)** — deferred (PRD §2.2 out of
  scope for v1). Noted as the intended direction when go-rag later supports
  non-loopback exposure.

## R4. REST contract + OpenAPI

**Decision**: Hand-author `contracts/rest-openapi.yaml` (3.1) describing the 11
endpoints and schemas; the REST adapter serves it at `GET /openapi.yaml`. No
generator dependency in v1 — kept in sync by the parity test (R6).

**Rationale**: OpenAPI gives curl/users a discoverable, tooling-friendly contract
(MuninnDB ships `openapi.yaml` the same way). Hand-authoring avoids a codegen dep;
the surface is small and stable.

**Alternatives considered**:
- **Generate OpenAPI from protobuf** (e.g. protoc-gen-openapi) — viable later if
  REST and gRPC schemas are unified; unnecessary for v1's hand-written REST.

## R5. Streaming semantics

**Decision**: **Unary-only for v1.** Query/Add/Status return complete responses.
MuninnDB streams activation results; go-rag's `index.Retrieval.Search` returns a
fully-materialized `[]Hit`, so streaming would be artificial. Add a **Server-Sent
Events** `GET /v1/events` (REST) and a streaming `Watch` RPC (gRPC) only if a
real consumer (e.g. live ingest progress) appears — deferred.

**Rationale**: Don't build streaming without a consumer. The async-after-ACK model
already decouples write ACK from indexing; status polling covers "indexing
pending." Keeps v1 simple.

## R6. Cross-transport parity guarantee (how FR-002/003 are met)

**Decision**: The `internal/engine` facade returns **structured result types**
(`QueryHit`, `StatusInfo`, `IngestSummary`, …). Each adapter (REST/gRPC/MCP) is a
pure serialization of those types — no independent logic. A single table-driven
test in `internal/engine` (or a cross-adapter test) runs each operation through
all three adapters and asserts byte-identical *structured* results.

**Rationale**: This is the MuninnDB "adapters, not separate implementations"
principle made testable. Parity is structural (shared facade) + verified (parity
test), not hoped-for. It also removes the current duplication where
`cli/wire.go` and `mcp/server.go` each reimplement `openDB`/`docOf`/`lookupChunk`/
`countPrefix` and the query/add/status wiring.

## R7. Port scheme (proposed, tasks-finalizable)

**Decision**: Distinct loopback ports, overridable per service, chosen to avoid
collision with MuninnDB (8474–8477/8750):

| Transport | Default addr | Flag |
|---|---|---|
| MCP (existing) | `127.0.0.1:7878` | `--mcp-addr` |
| REST (new) | `127.0.0.1:7879` | `--rest-addr` |
| gRPC (new) | `127.0.0.1:7880` | `--grpc-addr` |

**Rationale**: Keeps MCP's existing `:7878`; adds a clean consecutive block.
Default host `127.0.0.1` (not `:7878` wildcard) for REST/gRPC to enforce
loopback (Principle I). Persisted via an extended `daemon.Addrs{MCP,REST,GRPC}`.

**Note**: the existing `serve`/`start` use `:7878` (all-interfaces bind). Part of
this work hardens MCP to `127.0.0.1:7878` too, for consistency with Principle I.
