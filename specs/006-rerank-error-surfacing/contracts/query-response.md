# Contract: Query Response â€” RerankFailed flag (H09)

> Phase 1 output. go-rag exposes a query operation over three transports (MCP, REST,
> gRPC) plus a CLI, all adapters over one `engine.Engine.Query`. This contract fixes how
> the new `RerankFailed` signal is rendered on each, so the **same** failing-reranker
> query reports the **same** verdict everywhere (FR-004, SC-003).
>
> Two distinct failure modes leave `engine.Query`, and they use **different** contract
> channels â€” this is intentional (research.md D2/D3):

| Failure mode | Contract channel | Caller-visible result |
|---|---|---|
| **Rerank failure** (FR-001/002) | `RerankFailed` **field** on a successful response | results returned (fallback order) + flag = `true` |
| **Retrieval failure on rerank path** (FR-009) | non-nil **error** (existing error path) | query error, no result object |

The field channel is specified below per transport. The error channel reuses the
existing per-transport error mapping (CLI: print + non-zero exit; REST: HTTP status +
message; gRPC: coded status; MCP: JSON-RPC error) â€” unchanged by this feature, identical
to how spec 005's `ErrEmbeddingMismatch` is mapped.

## Field semantics (identical on every transport)

`RerankFailed` is a boolean:

- `true` â€” reranking was configured for this query, was attempted, and **failed**
  (rerank error or score-count mismatch). Results are present but in **fallback (RRF)
  order**, not reranked order.
- `false` / absent â€” reranking succeeded, was not configured, or had an empty candidate
  pool. These non-failure cases are **not** distinguished on the wire.

A consumer that wants fail-closed behavior treats `RerankFailed=true` as an error in its
own tooling (spec Assumptions).

## REST (`:7879`)

`GET/POST /v1/query` response gains one additive key:

```json
{
  "hits": [ { "chunk_id": "...", "document_id": "...", "score": 0.83, "content": "...",
              "file_path": "...", "page": 0 } ],
  "rerank_failed": false
}
```

- Type: `queryResponse.RerankFailed bool`, JSON tag `"rerank_failed"` (`internal/rest/types.go`).
- Mapping: `handleQuery` sets it from `engine.QueryResult.RerankFailed`
  (`internal/rest/engine_adapter.go`).
- **Backward compatible** â€” extra key; existing consumers ignore unknown keys.

## gRPC (`:7880`)

`Gorag/Query` response message gains one additive field:

```proto
message QueryResponse {
  repeated QueryHit hits = 1;
  bool rerank_failed = 2;          // true iff reranking was attempted and failed (H09)
}
```

- Mapping: `Adapter.Query` sets it from `engine.QueryResult.RerankFailed`
  (`internal/grpc/engine_adapter.go`).

- `proto/gen/` regenerated by the project's protoc/buf step.

## MCP (`:7878`, `go_rag_query`)

The MCP tool returns text. `renderQuery` prepends one warning line when
`RerankFailed=true`, before the hits:

```text
âš  reranking failed; showing fallback-ordered results (reranker may be down or mismatched)

- (score 0.812) <hit preview>
- (score 0.770) <hit preview>
```

- No change to the result list format.
- Additive: consumers that don't recognize the line are unaffected.

## CLI (`go-rag query`)

stdout remains the JSON result array (unchanged shape, so `jq` pipelines don't break).
The failure signal goes to **stderr** so it is visible to humans without polluting the
JSON contract:

```text
warning: reranking failed; results are in fallback order (see log for details)
[ { "source": "...", "page": 0, "score": 0.81, "chunk": "..." }, ... ]
```

- Stderr warning emitted only when `engine.QueryResult.RerankFailed=true`.

## Parity acceptance (FR-004 / SC-003)

A single failing-reranker query, issued over REST, gRPC, MCP, and CLI, must report the
failure on **all four** (`rerank_failed=true` / proto field / warning line / stderr line),
and a healthy-reranker query over all four must report **none**. This is asserted by
extending `internal/engine/parity_test.go`.
